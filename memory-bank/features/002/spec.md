# Spec: Алгоритм матчинга пользователей в Pod

Ссылка на brief: memory-bank/features/002/brief.md
Ссылка на issue: https://github.com/snblack/orbit/issues/1

## Цель

Сформировать Pod-ы из совместимых и географически близких пользователей, завершивших онбординг, и уведомить каждого участника о попадании в группу.

---

## Scope

**Входит:**
- Сервис матчинга `MatchingService` с детерминированным алгоритмом
- Модели `Pod`, `PodMembership`, `Notification` и их миграции
- Rake task `matching:run`
- RSpec-тесты на всю логику матчинга

**НЕ входит:**
- UI для просмотра Pod-а пользователем
- Push/email уведомления (только in-app запись в БД)
- AI-матчинг
- Автоматический запуск по расписанию

---

## Требования

### Модели и схема

1. Создать таблицу `pods`: `id`, `status` (string, `active` | `inactive`), `created_at`, `updated_at`.
2. Создать таблицу `pod_memberships`: `id`, `pod_id` (FK → pods), `user_id` (FK → users), `created_at`.
3. Создать таблицу `notifications`: `id`, `user_id` (FK → users), `pod_id` (FK → pods), `message` (string), `read` (boolean, default: false), `created_at`.
4. Модель `User` уже содержит поля `life_phase`, `social_style`, `friendship_goal`, `openness_level`, `social_frequency`, `schedule_preference` (jsonb), `latitude`, `longitude`, `onboarding_completed` — новых миграций на `users` не требуется.

### Алгоритм совместимости

5. Сравниваются 6 полей: `life_phase`, `social_style`, `friendship_goal`, `openness_level`, `social_frequency`, `schedule_preference`.
6. Поля `life_phase`, `social_style`, `friendship_goal`, `openness_level`, `social_frequency` — по 1 баллу за совпадение значений. `nil` не считается совпадением ни с каким значением, включая `nil` (даёт 0 баллов).
7. Поле `schedule_preference` (jsonb-хеш, ключи вида `{день}_{время}`):
   - `nil` трактуется как `{}` (пустой хеш)
   - N = количество общих ключей между двумя пользователями
   - N = 0 → 0 баллов
   - N ≥ 1 → балл = 1 + (N − 1) × 0.2
8. Минимальный порог совместимости для включения в Pod: суммарный score ≥ 3.

### Алгоритм геолокации

9. Базовый радиус поиска кандидатов для анкора — 5 км.
10. Если совместимых кандидатов в текущем радиусе менее 4 (не считая анкора) — радиус увеличивается на 5 км. Поиск на новом шаге ведётся по всему кругу целиком. Шаги: 5 → 10 → 15 → 20 → 25 → 30 км.
11. Расстояние считается по формуле Haversine.

### Порядок формирования Pod

12. Пул = пользователи с `onboarding_completed: true`, не состоящие ни в одном Pod, с непустыми `latitude` и `longitude`, упорядоченные по `id ASC`. Пользователи с `latitude: nil` или `longitude: nil` исключаются из пула наравне с пользователями без `onboarding_completed`.
13. Из пула берётся первый пользователь — анкор.
14. Среди оставшихся в пуле ищутся кандидаты: score ≥ 3 и в пределах текущего радиуса.
15. Если найдено ≥ 4 кандидатов — выбираются 4 с наивысшим score; при равном score — с наименьшим `id`. Анкор + 4 кандидата = Pod из 5 со статусом `active`.
16. Если при радиусе 30 км кандидатов < 4 — анкор + все найденные (0–3) образуют Pod со статусом `inactive`.
17. Все участники сформированного Pod-а (active или inactive) исключаются из пула.
18. Итерация повторяется до исчерпания пула.
19. Пользователи, оставшиеся в пуле после завершения всех итераций (< 5 человек), объединяются в один Pod со статусом `inactive`.

### Уведомления

20. В том же методе, где создаётся `PodMembership`, последовательно создаётся `Notification` с `message: "Вы вошли в группу"` и `read: false`. Оборачивать в отдельную транзакцию не нужно.

### Rake task

21. Команда: `rails matching:run`.
22. Идемпотентен: пользователи, уже состоящие в любом Pod, пропускаются.
23. Если подходящих пользователей < 5 — task завершается без ошибок, выводит `"Not enough users to form a pod."`, Pod-ов не создаёт.
24. При падении на середине выполнения уже созданные Pod-ы и уведомления сохраняются; при повторном запуске их участники пропускаются.

---

## Инварианты

- Пользователь одновременно состоит не более чем в одном Pod.
- В матчинг попадают только пользователи с `onboarding_completed: true`.
- Pod со статусом `active` содержит ровно 5 участников.
- Pod со статусом `inactive` содержит от 1 до 4 участников.
- Каждый участник любого Pod-а имеет ровно одну запись `Notification` на этот Pod.
- Порядок обработки пользователей детерминирован: всегда по `id ASC`.

---

## Acceptance Criteria

- [ ] `rails matching:run` формирует Pod-ы ровно из 5 участников со статусом `active`
- [ ] В Pod попадают только пользователи с `onboarding_completed: true`, не состоящие в Pod
- [ ] Два пользователя попадают в один Pod только при score ≥ 3
- [ ] `schedule_preference`: первое общее окно = 1 балл, каждое следующее = +0.2 балла
- [ ] Радиус расширяется пошагово 5 → 10 → ... → 30 км; за 30 км не выходит
- [ ] Если при 30 км кандидатов < 4 — анкор и найденные образуют Pod со статусом `inactive`
- [ ] Остаток пула (< 5 пользователей) объединяется в Pod со статусом `inactive`
- [ ] При равном score выбирается кандидат с меньшим `id`
- [ ] Каждый участник Pod-а получает запись в `notifications` с текстом "Вы вошли в группу"
- [ ] При < 5 подходящих пользователей task выводит `"Not enough users to form a pod."` и завершается без ошибок
- [ ] Повторный запуск task не создаёт дублей — участники существующих Pod-ов пропускаются
- [ ] Все существующие тесты проходят
- [ ] Инварианты не нарушены

---

## Ограничения

- Реализация — чистый Ruby-сервис, без AI и внешних зависимостей
- Haversine реализуется вручную в теле `MatchingService`. Не использовать гемы `geocoder`, `geokit` и аналоги
- Не использовать транзакции с откатом при падении task (поведение при сбое — сохранить созданное)
- Тесты — только RSpec; фабрики через FactoryBot
- Не менять существующую схему таблицы `users`
- Не менять логику онбординга
