# Brief: Алгоритм матчинга пользователей в Pod

Ссылка на задачу: https://github.com/snblack/orbit/issues/1

---

## Проблема

Пользователи проходят онбординг и заполняют профиль: жизненная фаза, стиль общения, ценности, ритм встреч, расписание, геолокация. Однако после завершения онбординга ничего не происходит — люди не попадают ни в какую группу и не могут познакомиться друг с другом. Продукт собирает данные, но не использует их по назначению.

Основная ценность Orbit — живые малые группы (Pod) из близких по духу людей рядом. Без механизма формирования Pod-ов продукт не выполняет свою главную функцию.

---

## Для кого

**Пользователь:** человек, завершивший онбординг и ожидающий попадания в группу. Он заполнил анкету, но не получает никакого продолжения — не знает, что дальше, не видит смысла возвращаться в приложение.

**Администратор:** запускает процесс формирования Pod-ов вручную (на этапе MVP) через Rake task.

---

## Контекст происхождения

Задача — первая фаза алгоритма матчинга в рамках проекта Orbit. Реализуется без AI, на детерминированной Ruby-логике. Данные для матчинга уже собираются через онбординг: `life_phase`, `social_style`, `friendship_goal`, `openness_level`, `social_frequency`, `schedule_preference`, `latitude`, `longitude`.

---

## Желаемый результат

Система может взять список пользователей, завершивших онбординг, и объединить их в Pod-ы по 5 человек с учётом профильной совместимости и географической близости. После формирования Pod-а каждый участник получает in-app уведомление о факте попадания в группу.

---

## Правила матчинга

**Совместимость по профилю (подсчёт совпадений):**
- Сравниваются поля: `life_phase`, `social_style`, `friendship_goal`, `openness_level`, `social_frequency`, `schedule_preference`
- Каждое совпавшее поле даёт 1 балл; все поля равнозначны
- Минимальный порог для включения в Pod: ≥3 баллов из 6
- Для `schedule_preference` (хеш вида `{"fri_evening" => "1", "sat_evening" => "1"}`): первое совпавшее общее ключ даёт 1 балл, каждое последующее общее ключ — 0.2 балла. Итоговый балл за поле = 1 + (N−1) × 0.2, где N — количество общих ключей (N ≥ 1); при N = 0 балл за поле = 0. Возможные ключи формируются как `{день}_{время}`, где день — `mon`, `tue`, `wed`, `thu`, `fri`, `sat`, `sun`; время — `morning`, `afternoon`, `evening`. Итого 21 возможный ключ.

**Географическая близость:**
- Базовый радиус: 5 км
- При нехватке совместимых пользователей (менее 4 кандидатов не считая анкора в текущем радиусе) радиус расширяется на 5 км. Поиск на каждом шаге ведётся по всему новому кругу целиком (не только в добавленном кольце). Расширение повторяется до достижения 30 км.
- Если при радиусе 30 км набрать 4 совместимых кандидата (не считая анкора) не удаётся — анкорный пользователь объединяется со всеми найденными кандидатами (0–3 человека) в Pod со статусом `inactive`. Все участники такого Pod-а исключаются из дальнейшего матчинга наравне с участниками активных Pod-ов.

**Размер Pod:**
- Ровно 5 участников
- Минимальный пул для запуска матчинга: 5 пользователей

**Порядок формирования Pod:**
1. Из пула подходящих пользователей берётся первый (по `id` ascending) — это анкор
2. Среди оставшихся ищутся кандидаты, удовлетворяющие порогу совместимости и радиусу (анкор + 4 кандидата = Pod из 5)
3. Если кандидатов ≥4 — выбираются 4 с наивысшим score совместимости (при равном score — с наименьшим `id`)
4. Сформированный Pod исключается из пула, процесс повторяется

---

## Граничные случаи

- **Остаток пользователей** — пользователи, оставшиеся в пуле после завершения всех итераций (их недостаточно для новой итерации): все они объединяются в один Pod со статусом `inactive`.
- **Нехватка кандидатов в итерации** — если для анкорного пользователя не удаётся набрать 4 совместимых кандидата даже при радиусе 30 км: анкор + все найденные кандидаты (0–3) образуют Pod со статусом `inactive` и исключаются из пула.
- **Пользователь уже в Pod:** исключается из следующего матчинга.

---

## Статус пользователя

Булев флаг `onboarding_completed` на модели `User` проставляется автоматически после последнего шага онбординга. В матчинг берутся только пользователи с `onboarding_completed: true`, не состоящие ни в одном Pod.

---

## Модель Pod

- Идентификатор: `id`
- Атрибут `status`: `active` | `inactive`
  - `active` — Pod из 5 участников, сформированный в пределах допустимого радиуса
  - `inactive` — Pod из менее чем 5 участников: либо остаток после завершения всех итераций, либо результат итерации, в которой анкор не набрал достаточно кандидатов даже при радиусе 30 км
- Связь с пользователями через join-таблицу `pod_memberships`

---

## Модель уведомлений

Таблица `notifications`, поля:
- `id`
- `user_id` (FK → users)
- `pod_id` (FK → pods)
- `message` (string) — фиксированный текст: "Вы вошли в группу"
- `read` (boolean, default: false)
- `created_at`

Канал: in-app (отображение на сайте). Уведомление создаётся одновременно с записью в `pod_memberships`.

---

## Интерфейс администратора

Rake task: `rails matching:run` — запускает полный цикл матчинга для всех подходящих пользователей. Идемпотентен: при повторном запуске пользователи, уже состоящие в любом Pod (активном или неактивном), пропускаются. Новые Pod-ы формируются только из пользователей, не состоящих ни в одном Pod на момент запуска.

При падении task на середине выполнения уже созданные Pod-ы и уведомления сохраняются (откат не производится). При повторном запуске их участники пропускаются.

Пользователи обрабатываются в порядке возрастания `id`. Это обеспечивает детерминированный результат при повторных запусках с одинаковым набором пользователей.

Если подходящих пользователей (`onboarding_completed: true`, не состоящих в Pod) меньше 5 — task завершается без ошибок, выводя сообщение `"Not enough users to form a pod."`

---

## Тесты

Вся логика матчинга покрывается тестами на RSpec:
- Сервис матчинга (совместимость, подбор по геолокации, формирование Pod-ов)
- Граничные случаи (недостаточно пользователей, расширение радиуса, остаток в `inactive` Pod)
- Создание уведомлений после формирования Pod-а

---

## Acceptance Criteria

1. Rake task `matching:run` формирует Pod-ы ровно из 5 участников.
2. В Pod попадают только пользователи с `onboarding_completed: true`.
3. Пользователь, уже состоящий в Pod, не участвует в новом матчинге.
4. Двое пользователей попадают в один Pod только если у них суммарный score совместимости ≥3. Пять полей (`life_phase`, `social_style`, `friendship_goal`, `openness_level`, `social_frequency`) дают по 1 баллу за совпадение. Для `schedule_preference` — первое общее окно даёт 1 балл, каждое последующее — 0.2 балла.
5. При нехватке кандидатов (менее 4 не считая анкора) в текущем радиусе он расширяется на 5 км. Расширение повторяется пошагово: 5 → 10 → 15 → 20 → 25 → 30 км. При каждом шаге поиск ведётся по всему новому кругу. Расширения за 30 км не происходит.
6. Пользователи, не вошедшие ни в один полный Pod, получают Pod со статусом `inactive` и исключаются из последующих циклов матчинга.
6a. Если для анкорного пользователя не найдено 4 совместимых кандидата даже при радиусе 30 км, анкор и все найденные кандидаты (сколько бы их ни было) образуют Pod со статусом `inactive`.
7. Каждый участник сформированного Pod-а получает запись в таблице `notifications` с сообщением о попадании в группу.
8. При равном score среди кандидатов в Pod попадает пользователь с меньшим `id`.
9. Если подходящих пользователей меньше 5, task завершается без ошибок и выводит `"Not enough users to form a pod."` — никаких Pod-ов не создаётся.
10. Вся логика матчинга покрыта тестами RSpec.
