# Pod Dashboard

**Issue:** #3

## Цель

Дать участнику Pod первый экран, где он видит свою группу: состав, историю и планы совместных активностей.

## Scope

- **Входит:**
  - Страница `/pods/:id` — Pod Dashboard
  - Страница `/pods/:pod_id/members/:user_id` — профиль участника
  - Модель `Activity` (`pod_id`, `occurred_on`, `status`, `proposed_by`)
  - Поле `User.display_name` (string, not null) + миграция с default
  - Поле `User.bio` (text, nullable, макс. 300 символов)
  - Шаг `profile` в онбординге (новый шаг; `display_name` обязательно, `bio` необязательно)
  - Inline-форма предложения активности через Turbo Frame
  - Ссылка «Мой Pod» в навигационном хедере
  - Seed с тестовыми данными для воспроизведения всех 8 критериев
  - Отображение существующих полей `User`: `life_phase`, `friendship_goal`, `interests` на странице профиля участника (поля уже присутствуют в модели из предыдущих шагов онбординга; в этой задаче только читаются, не изменяются)

> **Примечание по scope:** спека затрагивает 6 модулей (`User`, `Onboarding`, `Activity`, Pod Dashboard, Member Profile, Navigation Header), что превышает стандартный лимит в 3. Разбивка намеренно отложена: все части атомарно связаны — без `display_name` нет аватаров, без шага `profile` нет `display_name`, без `Activity` нет Dashboard. Отдельные задачи создадут лишние миграции и незавершённые экраны.

- **НЕ входит:**
  - Загрузка аватаров (вместо них — инициалы на цветном фоне)
  - Уведомления и механизм доставки (уже реализованы; URL ведёт на `/pods/:id`)
  - Алгоритм матчинга (реализован в issue #1)
  - Редактирование профиля после онбординга
  - Удаление или редактирование активностей

## Требования

1. `User` получает поле `display_name` (string, not null); default при миграции — часть email до `@`.
2. `User` получает поле `bio` (text, nullable); валидация `length: { maximum: 300 }` на уровне модели.
3. Шаг `profile` онбординга собирает `display_name` (обязательно) и `bio` (необязательно). При попытке перейти к следующему шагу с пустым `display_name` форма показывает inline-ошибку; шаг не меняется.
4. Создаётся модель `Activity` с полями `pod_id`, `occurred_on` (date), `status` (`completed` | `planned`), `proposed_by` (user_id, FK на `users`).
5. `Activity` валидирует `occurred_on >= Date.today` для записей со статусом `planned` при создании.
6. Dashboard `/pods/:id` для активного Pod показывает: список участников (каждый участник — ссылка на `/pods/:pod_id/members/:user_id`), номер текущей недели, последнюю завершённую активность, ближайшую запланированную активность (если есть), кнопку «Предложить активность».
7. Dashboard для неактивного Pod показывает: список участников и плашку «Группа ещё формируется» — блоки активностей (включая заглушку «Ещё не встречались»), блок ближайшей активности и кнопка «Предложить активность» не отображаются и отсутствуют в DOM.
8. Инициалы-аватар: первые буквы первых двух слов `display_name`; цвет фона `hsl((user.id * 37) % 360, 55%, 65%)`, детерминирован.
9. Номер недели: `(Date.today - pod.created_at.to_date).to_i / 7 + 1`.
10. Дата активности отображается через `I18n.l(activity.occurred_on, format: :long)`; формат `:long` определён в `ru.yml` и даёт «5 апреля 2026».
11. Последняя завершённая активность: если нет — отображается «Ещё не встречались» (блок присутствует в DOM).
12. Ближайшая запланированная активность: если нет — блок не рендерится в HTML (отсутствует в DOM).
13. Форма предложения активности открывается inline через `<turbo-frame id="new_activity_form">`; кнопка скрывается, пока фрейм открыт или загружается. Во время загрузки фрейм остаётся пустым — без спиннера или скелетона (стандартное поведение Turbo).
14. При ошибке валидации формы контроллер отвечает `422` и рендерит форму заново с ошибкой; Turbo обновляет только фрейм. При ошибке сервера (5xx) контроллер устанавливает `flash[:alert]`, который рендерится в `application.html.erb` стандартным partial `_flash.html.erb`; форма не закрывается, кнопка остаётся скрытой. 5xx-сценарий покрывается controller-тестом (stub на `Activity.save` → raise `ActiveRecord::StatementInvalid`); acceptance criteria для ручного теста не предусмотрен.
15. После успешного создания активности — `redirect_to pod_path(@pod)`; в блоке «ближайшая активность» отображается запланированная активность с минимальным `occurred_on >= Date.today` (не обязательно только что созданная).
16. Страница профиля участника доступна только членам Pod; не-членам — 404.
17. Прямой переход на `/pods/:id` чужого Pod — 404.
18. Ссылка «Мой Pod» в хедере видна авторизованным пользователям с любым Pod; ведёт на последний по `created_at`. Нет Pod — ссылка не отображается.
19. Неавторизованный пользователь перенаправляется на страницу входа (Devise `authenticate_user!`).

## Инварианты

- Просмотр Dashboard и профилей участников доступен только членам конкретного Pod — чужой Pod возвращает 404.
- `display_name` не может быть пустым ни у одного пользователя после миграции.
- `Activity.occurred_on` для статуса `planned` всегда `>= Date.today` на момент создания.
- Цвет инициалов-аватара детерминирован по `user.id` и не изменяется между запросами.
- Неавторизованный пользователь не получает доступ ни к одному Pod-маршруту.

## Acceptance Criteria

- [ ] Пользователь `alice@example.com` после `rails db:seed` входит и видит Dashboard активного Pod с двумя участниками
- [ ] Seed создаёт Pod с `created_at = 14.days.ago`; на Dashboard отображается «Неделя 3»
- [ ] Блок «последняя активность» показывает дату завершённой активности в формате «5 апреля 2026»
- [ ] Блок «ближайшая активность» показывает дату запланированной активности
- [ ] Кнопка «Предложить активность» открывает inline-форму; кнопка скрывается
- [ ] Форма с прошедшей датой возвращает ошибку inline, не перезагружает страницу полностью
- [ ] Успешное создание активности обновляет страницу; в блоке «ближайшая активность» отображается запланированная активность с минимальным `occurred_on >= Date.today`
- [ ] Клик по участнику открывает `/pods/:pod_id/members/:user_id` с его `display_name`, `bio`, `life_phase`, `friendship_goal`, `interests`
- [ ] Пользователь неактивного Pod видит список участников и плашку «Группа ещё формируется»; блок «Ещё не встречались», блок ближайшей активности и кнопка «Предложить активность» отсутствуют в DOM
- [ ] Переход на чужой Pod возвращает 404
- [ ] Неавторизованный переход на `/pods/:id` редиректит на страницу входа
- [ ] Ссылка «Мой Pod» отображается в хедере у пользователя с Pod и отсутствует без Pod
- [ ] Онбординг-шаг `profile` сохраняет `display_name` и `bio`; `display_name` обязательно, `bio` необязательно
- [ ] Попытка перейти к следующему шагу онбординга с пустым `display_name` показывает inline-ошибку; шаг не меняется
- [ ] `bio` пустого участника не рендерится в HTML карточки
- [ ] Все существующие тесты проходят

## Ограничения

- Использовать Turbo Frames для inline-формы активности — не заменять на JS-решения
- Аватары не реализовывать; только инициалы с детерминированным цветом по формуле `hsl((id * 37) % 360, 55%, 65%)`
- Human-readable значения онбординг-полей брать только через I18n (`t("onboarding.life_phase.#{value}")` и аналогично)
- Формат даты `:long` определить в `config/locales/ru.yml`, не хардкодить строку
- Авторизацию строить на стандартном Devise `authenticate_user!` и проверке членства в Pod
- Не добавлять редактирование или удаление активностей
