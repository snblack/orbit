# Brief: Pod Dashboard

Ссылка на задачу: https://github.com/snblack/orbit/issues/2

---

## Проблема

После того как пользователь попал в Pod (через алгоритм матчинга), у него нет места в приложении, где он мог бы видеть свою группу. Он не знает, кто ещё в его Pod, на какой неделе существования группы они находятся, с кем и когда виделись последний раз, и что запланировано дальше. Пользователь получил уведомление о попадании в группу — но дальше тупик: некуда идти, нечего делать.

---

## Для кого

**Участник активного Pod:** прошёл онбординг и матчинг, попал в Pod со статусом `active`. Хочет познакомиться с группой, понять, кто эти люди, и видеть общий контекст — историю и планы совместных активностей.

**Участник неактивного Pod:** попал в Pod со статусом `inactive` (группа не укомплектована). Видит Dashboard в ограниченном режиме: только список участников и статус «группа ещё формируется». Блоки «последняя активность», «ближайшая активность» и кнопка «предложить встречу» не отображаются.

---

## Контекст происхождения

Это следующий шаг после фичи матчинга (issue #1). Алгоритм формирует Pod-ы, но после формирования пользователь оказывается в вакууме: нет страницы группы, нет профилей участников, нет хронологии. Pod Dashboard — первый экран, который делает Pod «живым» в интерфейсе.

**Что создаётся в рамках этой задачи:**

- Модель `Activity` с атрибутами: `pod_id`, `occurred_on` (date), `status` (`completed` | `planned`), `proposed_by` (user_id). Других обязательных атрибутов нет.
- Поле `User.display_name` (string, not null) — отображаемое имя участника. Добавляется миграцией; на шаге `profile` онбординга добавляется как отдельное обязательное поле (email остаётся для аутентификации и не убирается). Шаг `profile` в онбординге **создаётся в рамках этой задачи** — ранее такого шага не существовало. При миграции существующим пользователям устанавливается default: часть email до `@`.
- Поле `User.bio` (text, nullable, макс. 300 символов) — короткое описание себя. Валидируется на уровне модели (`validates :bio, length: { maximum: 300 }`). Необязательное; если пусто — блок bio на карточке не рендерится в HTML.
- Аватар не реализуется в этой задаче. Вместо аватара — инициалы из `display_name` на цветном фоне: берутся первые буквы первых двух слов `display_name` (например, `"John Doe"` → `"JD"`, `"John Michael Doe"` → `"JM"`, `"Alice"` → `"A"`). Цвет фона детерминирован по `user.id` (`hsl((id * 37) % 360, 55%, 65%)`), не меняется при перезагрузке.

**Что уже существует и используется как есть:**

- `Pod` с полями `status` (`active` | `inactive`) и `created_at`.
- `PodMembership` связывает `User` и `Pod`.
- Данные онбординга (`life_phase`, `friendship_goal`, `interests`) отображаются на странице профиля участника — никакой дополнительной модели не нужно. Human-readable значения этих полей берутся из I18n: ключи вида `t("onboarding.life_phase.#{value}")`. Поле `interests` хранится как массив строк (`string[]`); при отображении каждый элемент переводится через `t("onboarding.interests.#{value}")` и выводится списком.

## Маршрутизация и доступ

- URL Dashboard: `/pods/:id`.
- Создание активности: `POST /pods/:pod_id/activities`. Обрабатывается `Activities::CreateController` (или вложенный `ActivitiesController` в пространстве `Pods`).
- Точка входа: ссылка в уведомлении о попадании в Pod (уже реализована через `Notification`; URL уведомления ведёт на `/pods/:id` — до выполнения этой задачи такой маршрут не существует, это ожидаемо); навигационная ссылка «Мой Pod» в хедере, видна авторизованным пользователям, у которых есть Pod с **любым** статусом (`active` или `inactive`). Если у пользователя несколько Pod-ов (технически возможно), ссылка ведёт на последний по `created_at`.
- Если у пользователя нет Pod — ссылка «Мой Pod» не отображается; прямой переход по `/pods/:id` Pod-а, членом которого пользователь не является, возвращает 404.
- Неавторизованный пользователь при переходе на `/pods/:id` или `/pods/:pod_id/members/:user_id` перенаправляется на страницу входа (стандартное поведение Devise `authenticate_user!`).
- Страница профиля участника: `/pods/:pod_id/members/:user_id`. Доступна только членам этого Pod; не-членам — 404.

## Желаемый результат

### Активный Pod (6 критериев)

1. **Список участников.** Отображаются все члены Pod: инициалы-аватар, `display_name`, `bio` (если заполнено, иначе блок bio не рендерится). Клик по участнику открывает его профиль.
2. **Профиль участника.** Страница `/pods/:pod_id/members/:user_id` показывает `display_name`, `bio`, `life_phase`, `friendship_goal`, `interests`. Значения `life_phase`, `friendship_goal`, `interests` отображаются как plain text через I18n (`t("onboarding.life_phase.#{value}")` и аналогично для остальных полей). Доступна только членам Pod.
3. **Номер текущей недели.** Вычисляется как `(Date.today - pod.created_at.to_date).to_i / 7 + 1`. Отображается как «Неделя N». Без пользовательского описания — только номер.
4. **Последняя совместная активность.** Последняя запись `Activity` с `status: completed`, отсортированная по `occurred_on` убыванию. Показывается дата через `I18n.l(activity.occurred_on, format: :long)` — формат `:long` должен быть определён в локальном файле `ru.yml` и давать результат вида «5 апреля 2026» (число + месяц в родительном падеже + год). Если активностей нет — отображается «Ещё не встречались» (блок присутствует в DOM).
5. **Ближайшая запланированная активность.** Первая запись `Activity` с `status: planned` и `occurred_on >= Date.today`, отсортированная по `occurred_on` возрастанию. Показывается дата через `I18n.l(activity.occurred_on, format: :long)` (тот же формат, что в п. 4). Если нет — блок не рендерится в HTML (отсутствует в DOM). Намеренное отличие от п. 4: для завершённых активностей плейсхолдер «Ещё не встречались» полезен как социальный сигнал; для запланированных — пустой блок визуально лишний.
6. **Предложить активность.** Кнопка открывает inline-форму через Turbo Frame (`<turbo-frame id="new_activity_form">`): форма появляется под кнопкой, кнопка скрывается, пока фрейм открыт. Форма содержит одно поле — дату встречи (`occurred_on`). Дата должна быть `>= Date.today`; валидация на уровне модели (`validates :occurred_on, comparison: { greater_than_or_equal_to: -> { Date.today } }`). При ошибке контроллер отвечает статусом `422 Unprocessable Entity` и рендерит форму заново с ошибкой inline под полем даты — Turbo Frame обновляет только фрейм. После успешного сохранения контроллер выполняет `redirect_to pod_path(@pod)` (Turbo выполняет полный redirect); страница перезагружается, созданная активность сразу отображается в п. 5. Создаётся `Activity` с `status: planned`, `proposed_by: current_user.id`.

### Неактивный Pod (2 критерия)

7. **Список участников** — отображается так же, как в активном Pod (п. 1), включая клик, открывающий профиль участника по `/pods/:pod_id/members/:user_id`.
8. **Статус группы** — вместо блоков активностей и кнопки отображается плашка «Группа ещё формируется».

## Критерий готовности

Задача считается выполненной, если все 8 пунктов воспроизводимы в браузере после выполнения `rails db:seed`.

Seed содержит:
- **Активный Pod** с двумя участниками: первый — `display_name: "Alice Green"`, с заполненным `bio`; второй — `display_name: "Bob Lee"`, `bio` пустой. Pod имеет одну завершённую и одну запланированную активность.
- **Неактивный Pod** с одним участником.
- Для входа в систему: email `alice@example.com`, пароль `password` — участник активного Pod.
