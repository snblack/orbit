# Редизайн форм авторизации Devise

**Brief:** `memory-bank/features/001/brief.md`
**Issue:** не создан (задача локальная)

## Цель
Заменить дефолтные Devise-формы на кастомные с полями ввода, кнопками и отображением ошибок в стиле онбординга проекта.

## Scope
- Входит: генерация и редизайн четырёх Devise-вью (`sessions/new`, `registrations/new`, `passwords/new`, `passwords/edit`)
- НЕ входит: контроллеры, маршруты, модели Devise; email-шаблоны; 2FA и расширения; обработка просроченного токена сброса пароля (Devise по умолчанию редиректит на `passwords/new` с flash-сообщением); чекбокс "Запомнить меня" (`remember_me`) — не включать в форму `sessions/new`

## Поля форм

| Форма                    | Поле                  | Label                   | Placeholder             |
|--------------------------|-----------------------|-------------------------|-------------------------|
| `sessions/new`           | email                 | Email                   | your@email.com          |
| `sessions/new`           | password              | Пароль                  | Минимум 6 символов      |
| `registrations/new`      | email                 | Email                   | your@email.com          |
| `registrations/new`      | password              | Пароль                  | Минимум 6 символов      |
| `registrations/new`      | password_confirmation | Подтверждение пароля    | Повторите пароль        |
| `passwords/new`          | email                 | Email                   | your@email.com          |
| `passwords/edit`         | password              | Новый пароль            | Новый пароль            |
| `passwords/edit`         | password_confirmation | Подтверждение пароля    | Повторите пароль        |

## Требования
1. Все четыре формы имеют кастомные ERB-файлы в `app/views/devise/`
2. Каждое поле содержит `<label>` и `placeholder` согласно таблице полей выше
3. Поля стилизованы в стиле онбординга: скруглённые, с серой рамкой и внутренними отступами; при фокусе — цветная обводка акцента
4. Кнопка submit в стиле акцентной кнопки проекта: фон цвета акцента, белый текст, скруглённая, с hover-эффектом затемнения
5. Блок ошибок валидации рендерится **только при** `resource.errors.any?` — на initial render (без ошибок) блок отсутствует в DOM. Структура:
   ```erb
   <% if resource.errors.any? %>
     <div class="...">  <!-- красный фон, красная рамка, красный текст, скруглённый, с отступами -->
       <ul><% resource.errors.full_messages.each do |msg| %><li><%= msg %></li><% end %></ul>
     </div>
   <% end %>
   ```
6. Межформенные ссылки в стиле акцентных ссылок проекта: цвет акцента, underline при hover, мелкий текст. Полный список ссылок по формам:

   | Форма                | Хелпер пути                  | Текст ссылки                        |
   |----------------------|------------------------------|-------------------------------------|
   | `sessions/new`       | `new_user_registration_path` | Нет аккаунта? Зарегистрироваться    |
   | `sessions/new`       | `new_user_password_path`     | Забыли пароль?                      |
   | `registrations/new`  | `new_user_session_path`      | Уже есть аккаунт? Войти             |
   | `passwords/new`      | `new_user_session_path`      | Вернуться к входу                   |
   | `passwords/edit`     | —                            | ссылок нет                          |
7. Контейнер формы ограничен по ширине и центрирован; внутри — карточка с белым фоном, скруглёнными углами, тенью и внутренними отступами
8. Label: мелкий текст, полужирный, тёмно-серый, с нижним отступом до поля

## Flash-сообщения
Flash-сообщения (`notice`, `alert`) рендерятся глобально в `app/views/layouts/application.html.erb` — дополнительный рендеринг внутри Devise-форм не нужен. Агент обязан проверить это допущение перед реализацией: если layout не рендерит flash, добавить блок над формой — `notice`: зелёный фон, зелёная рамка, зелёный текст, скруглённый; `alert`: красный фон, красная рамка, красный текст, скруглённый.

## Инварианты
- Логика аутентификации Devise не изменяется (контроллеры, хелперы, маршруты остаются нетронутыми)
- Все `form_for`/`form_with` используют Devise-хелперы (`resource`, `resource_name`) — серверная часть форм не переписывается
- CSRF-токены присутствуют во всех формах (обеспечивается стандартными Rails-хелперами)

## Acceptance Criteria
- [ ] `app/views/devise/sessions/new.html.erb` — файл существует
- [ ] `app/views/devise/registrations/new.html.erb` — файл существует
- [ ] `app/views/devise/passwords/new.html.erb` — файл существует
- [ ] `app/views/devise/passwords/edit.html.erb` — файл существует
- [ ] Каждое поле имеет `<label>` и `placeholder` со значениями из таблицы полей выше (проверяется grep по ERB)
- [ ] Блок ошибок рендерится условно (`resource.errors.any?`) и отсутствует в DOM при отсутствии ошибок (проверяется grep: наличие `if resource.errors.any?` в каждом файле)
- [ ] Ни один элемент формы не выходит за границы viewport при ширине 320px (нет горизонтального скролла, контент не обрезается) — проверяется system spec (`bundle exec rspec spec/system/`) или вручную через DevTools
- [ ] Flash-сообщения отображаются корректно: либо через глобальный layout, либо через блок внутри формы (если layout не рендерит flash)
- [ ] Все существующие тесты проходят (`bundle exec rspec spec/ --exclude-pattern "**/system/**"`)
- [ ] Инварианты не нарушены

## Ограничения
- Не добавлять новые npm-пакеты и не расширять `tailwind.config.js`
- Не менять `devise.rb`, роуты и модель `User`
- Файл `app/views/layouts/application.html.erb` не изменяется (за исключением добавления flash-блока, если он отсутствует, согласно разделу Flash-сообщения)
- Не генерировать лишние Devise-вью (mailer, shared) — только четыре целевые формы
