# Realtime чат внутри Pod

**Source:** `memory-bank/features/004/brief.md`

## Цель

Дать участникам Pod возможность общаться в реальном времени прямо на Dashboard, не покидая приложение.

## Scope

- **Входит:**
  - Модель `Message` (pod_id, user_id, body, created_at)
  - `PodChannel` (ActionCable) — subscribe/broadcast сообщений
  - `MessagesController` — `POST /pods/:pod_id/messages` (создание сообщения через HTTP)
  - Изменение `ApplicationCable::Connection` — идентификация текущего пользователя
  - Секция чата на `/pods/:id` через Turbo Streams
  - Optimistic update для отправителя
  - История: последние 50 сообщений, infinite scroll вверх порциями по 50
  - Счётчик символов (лимит 1000), блокировка отправки при превышении
  - Режим read-only для Pod со статусом `inactive`
  - Авторизация на уровне сервера: только участник Pod может подписаться на канал и отправлять сообщения

- **НЕ входит:**
  - Личные сообщения между участниками
  - Удаление/редактирование сообщений
  - Индикатор статуса соединения WebSocket в UI
  - Offline-очередь исходящих сообщений
  - Статус `disbanded` у Pod (в текущей модели его нет — brief опережает модель; реализуем только для `active`/`inactive`)
  - Push-уведомления вне браузера

## Требования

1. Участник активного Pod может отправить текстовое сообщение; оно появляется у всех участников без перезагрузки страницы в течение ≤ 1 секунды.
2. Каждое сообщение отображает display_name отправителя и время отправки: `HH:MM` (если сегодня) или `DD.MM.YYYY HH:MM` (старше суток).
3. Максимальная длина сообщения — 1000 символов. Счётчик `X / 1000` отображается всегда; становится красным при превышении. Кнопка «Отправить» неактивна при пустом поле или превышении лимита.
4. `textarea`: Enter → отправить, Shift+Enter → перенос строки. После успешной отправки поле очищается.
5. Optimistic update: сообщение отправителя добавляется в список немедленно, до подтверждения сервера. Pending-сообщение отображается с классом `.message--pending` (Tailwind: `opacity-50`). Broadcast от сервера для отправителя игнорируется (без дублирования): сервер включает в broadcast-payload поле `sender_id` (id текущего пользователя); Stimulus-контроллер сравнивает его с `meta[name="current-user-id"]` — если совпадает, Turbo Stream игнорируется. После HTTP 200 pending-элемент остаётся в DOM, с него снимается класс `.message--pending` (реальный id и серверный timestamp не подставляются). При ошибке (HTTP 4xx/5xx, таймаут ≥ 10 секунд (fetch AbortController), срабатывание `subscription.rejected` callback в ActionCable) — optimistic-сообщение помечается классом `.message--failed` (красный текст + иконка ⚠). Под текстом `.message--failed` в отдельной строке появляется кнопка «Повторить». При клике существующая pending-запись заменяется новым optimistic-сообщением, запрос отправляется повторно. Ошибки 403/422 не ретраятся — вместо кнопки показывается текст ошибки: Stimulus читает `response.json()` и берёт поле `error`.
6. При открытии чата загружаются последние 50 сообщений (хронологически, старые вверху). Пока Turbo Frame загружается, внутри секции чата отображается spinner (аналогично spinner при пагинации). Если сообщений ≤ 50 — кнопки «загрузить ещё» нет. Если сообщений нет — отображается текст «Пока нет сообщений. Начните общение!». При ошибке начальной загрузки (Turbo Frame вернул ошибку) в секции чата отображается «Не удалось загрузить историю» с кнопкой «Повторить», перезагружающей Turbo Frame.
7. Когда первое загруженное сообщение становится видимым во viewport (Intersection Observer, threshold: 0.1), подгружается предыдущая порция из 50 (где «первое» — самое старое из текущих отображаемых). Во время загрузки вверху списка показывается spinner; триггер заблокирован до завершения запроса. Позиция прокрутки не сбрасывается (prepend новых элементов компенсируется корректировкой `scrollTop`). При ошибке загрузки — показывается inline-сообщение «Не удалось загрузить сообщения» с кнопкой «Повторить».
8. Новый участник Pod видит всю историю с момента создания Pod.
9. Сервер проверяет членство при `subscribe` на `PodChannel` и при каждом `speak`/`send_message`. Невалидная подписка отклоняется.
10. Секция чата не отображается пользователю, не являющемуся участником Pod (контроллер уже отдаёт 404 для чужих Pod — дополнительная проверка не требуется).
11. Pod со статусом `inactive`: поле ввода и кнопка «Отправить» скрыты, счётчик символов скрыт, история доступна (read-only).

## Инварианты

- Пользователь может читать/писать только в чат своего Pod (`pod_memberships` проверяется на сервере при каждом действии).
- Сообщения не удаляются при смене статуса Pod.
- Счётчик символов всегда отражает актуальную длину поля ввода.
- Broadcast никогда не дублирует optimistic-сообщение для отправителя.
- `display_name` — метод `User#display_name` (уже существует в модели); именно он используется в разметке сообщения.

## Acceptance Criteria

- [ ] Участник отправляет сообщение — оно появляется в его ленте без перезагрузки (optimistic).
- [ ] Второй участник того же Pod получает сообщение без перезагрузки в течение ≤ 1 секунды.
- [ ] Участник другого Pod не видит сообщений чужого чата через UI.
- [ ] Прямое подключение к `PodChannel` чужого Pod отклоняется сервером.
- [ ] При открытии чата отображаются последние 50 сообщений.
- [ ] При отсутствии сообщений отображается текст «Пока нет сообщений. Начните общение!».
- [ ] При прокрутке к самому старому из отображаемых сообщений подгружается предыдущая порция; во время загрузки виден spinner; позиция прокрутки не прыгает.
- [ ] При ошибке начальной загрузки отображается «Не удалось загрузить историю» с кнопкой «Повторить».
- [ ] При ошибке пагинации отображается inline-сообщение с кнопкой «Повторить».
- [ ] Новый участник видит всю историю с момента создания Pod.
- [ ] Отправка сообщения > 1000 символов заблокирована; счётчик красный.
- [ ] Кнопка «Отправить» неактивна при пустом поле.
- [ ] После успешной отправки поле ввода очищается.
- [ ] Enter отправляет, Shift+Enter — перенос строки.
- [ ] При ошибке отправки optimistic-сообщение помечается `.message--failed`; появляется кнопка «Повторить»; повторная отправка заменяет pending-запись новым optimistic-сообщением.
- [ ] Ошибки 403/422 не показывают кнопку «Повторить» — только текст ошибки.
- [ ] Чат Pod со статусом `inactive` открывается в read-only: поле ввода, кнопка и счётчик скрыты.
- [ ] Все существующие тесты проходят.

## Порядок реализации

1. `Message` — модель + миграция
2. `ApplicationCable::Connection` — идентификация пользователя
3. `PodChannel` + `MessagesController` — канал и HTTP-эндпоинт
4. `pods/show` — секция чата, Turbo Streams, Turbo Frame
5. `chat_controller.js` — Stimulus: optimistic update, счётчик символов, infinite scroll

## Ограничения

- Использовать ActionCable + Turbo Streams (Hotwire) — не вводить сторонних WebSocket-библиотек.
- Не создавать отдельный route для чата — только секция на `/pods/:id`.
- Авторизацию реализовывать через проверку `current_user` в `ApplicationCable::Connection` и членства в `PodChannel#subscribed`.
- Не менять логику `PodsController#set_pod` (авторизация по Pod уже работает через `current_user.pods`).
- `MessagesController#create`: маршрут `POST /pods/:pod_id/messages`, принимает `{ message: { body: "..." } }`. Успех — `200 OK` (пустой body). Ошибка авторизации — `403 JSON { "error": "текст" }`. Ошибка валидации — `422 JSON { "error": "текст" }`.
- Stimulus читает текст ошибки через `response.json().then(data => data.error)` и вставляет как `textContent` (не innerHTML).
- Стили — Tailwind (как во всём проекте).
- Весь клиентский JS — один Stimulus-контроллер `chat_controller.js`; логика optimistic update, счётчика символов и infinite scroll — в нём.
