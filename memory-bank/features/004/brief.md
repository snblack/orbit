# Brief: Realtime чат внутри Pod

Ссылка на задачу: https://github.com/snblack/orbit/issues/3

---

## Проблема

Участники Pod не могут общаться внутри приложения. После того как группа сформирована и Dashboard отображает её состав, у людей нет способа написать друг другу, договориться о встрече или обсудить общие интересы, не покидая платформу. Общение происходит в сторонних мессенджерах, что разрывает контекст и снижает вовлечённость в Pod.

---

## Для кого

**Участник активного Pod:** хочет написать своей группе, задать вопрос, договориться о следующей встрече или просто познакомиться — не выходя из приложения.

**Участник неактивного или завершённого Pod:** чат доступен в режиме только для чтения — новые сообщения отправить нельзя, история остаётся доступной.

---

## Контекст происхождения

Это следующий шаг после Pod Dashboard (issue #2). Dashboard сделал Pod «видимым» — теперь нужно сделать его «живым»: добавить коммуникацию в реальном времени. Задача помечена как `1.5`, то есть она является дополнением к основному циклу фичей первой версии и логически продолжает Dashboard.

**Технический контекст:** в приложении уже используется Turbo (Hotwire), поэтому ActionCable и Turbo Streams — естественный выбор для realtime-обновлений без перезагрузки страницы.

---

## Желаемый результат

### Размещение в интерфейсе

Чат располагается на странице Pod Dashboard (`/pods/:id`) как отдельная секция ниже списка участников. Отдельный route для чата не создаётся.

### Групповой чат

- Участники Pod могут отправлять и получать **текстовые сообщения** в рамках общего чата своей группы. Личные сообщения между участниками в Phase 1 не поддерживаются.
- Новое сообщение появляется у всех участников без перезагрузки страницы в течение **не более 1 секунды** при нормальных условиях сети (широкополосное соединение, latency ≤ 100ms).
- Максимальная длина одного сообщения — **1000 символов**. Отправка сообщения сверх лимита блокируется: кнопка отправки неактивна, под полем ввода отображается счётчик оставшихся символов (пример: `«950 / 1000»`). Счётчик отображается всегда, становится красным при превышении лимита.
- Каждое сообщение отображает **имя отправителя** и **время отправки** (формат: `HH:MM`, для сообщений старше суток — `DD.MM.YYYY HH:MM`).
- Собственное сообщение пользователь видит сразу после отправки (optimistic update); при ошибке доставки сообщение помечается как недоставленное с возможностью повторить отправку. Broadcast, пришедший от сервера для отправителя, игнорируется — сообщение уже отображено optimistically и не должно дублироваться.
- После успешной отправки поле ввода очищается автоматически.
- Поле ввода — многострочный `textarea`. **Enter** отправляет сообщение, **Shift+Enter** добавляет перенос строки. Кнопка «Отправить» неактивна, если поле пустое или длина текста превышает 1000 символов.

### История сообщений

- При открытии чата загружаются последние **50 сообщений** в хронологическом порядке (старые вверху, новые внизу). Если сообщений меньше 50 — загружаются все доступные без кнопки "загрузить ещё".
- Когда пользователь прокручивает список вверх и **первое загруженное сообщение становится видимым во viewport** (через Intersection Observer), автоматически подгружается предыдущая порция из 50 сообщений (паттерн "load more on scroll"). Во время загрузки отображается индикатор; позиция прокрутки сохраняется — список не прыгает.
- Новый участник, вступивший в Pod, видит всю историю сообщений с момента создания Pod.

### Доступ и изоляция

- Чат видят только участники Pod. Пользователь, не являющийся участником, не видит секцию чата на Dashboard.
- Сервер проверяет членство при subscribe на `PodChannel` и отклоняет подписку, если пользователь не является участником данного Pod. Клиентская проверка не является достаточной.
- Пользователь не может получить сообщения чужого Pod ни через UI, ни через прямое подключение к каналу.

### Разрыв соединения

- При потере WebSocket-соединения ActionCable выполняет автоматический reconnect штатными средствами (без дополнительной логики на клиенте).
- Сообщения, отправленные другими участниками во время разрыва, подгружаются при восстановлении соединения через стандартный механизм Turbo Streams.
- Реализация offline-очереди исходящих сообщений и явных UI-индикаторов состояния соединения — вне скоупа Phase 1.

### Жизненный цикл

Поведение чата определяется статусом Pod:

| Статус Pod | Отправка сообщений | Просмотр истории |
|---|---|---|
| `active` | доступна | доступна |
| `inactive` | недоступна | доступна |
| `disbanded` | недоступна | доступна |

- После перехода Pod в статус `inactive` или `disbanded` интерфейс ввода скрывается, чат открывается в режиме только для чтения.
- Сообщения не удаляются при смене статуса Pod.
- Удаление сообщений в Phase 1 не поддерживается.

### Критерии готовности (Definition of Done)

- [ ] Участник может отправить сообщение и увидеть его в ленте без перезагрузки страницы.
- [ ] Второй участник того же Pod получает это сообщение без перезагрузки в течение 1 секунды.
- [ ] Участник другого Pod не видит сообщений чужой группы через UI.
- [ ] Сервер отклоняет subscribe на `PodChannel` чужого Pod — прямое подключение к каналу невозможно.
- [ ] Не-участник Pod не видит секцию чата на Dashboard.
- [ ] При открытии чата отображаются последние 50 сообщений (или все, если их меньше).
- [ ] При прокрутке вверх до первого загруженного сообщения подгружается предыдущая порция; позиция прокрутки не сбрасывается.
- [ ] Новый участник Pod видит всю историю с момента создания Pod.
- [ ] Отправка сообщения длиннее 1000 символов заблокирована; счётчик символов становится красным.
- [ ] Кнопка «Отправить» неактивна при пустом поле ввода.
- [ ] После успешной отправки поле ввода очищается.
- [ ] Enter отправляет сообщение, Shift+Enter добавляет перенос строки.
- [ ] Чат Pod со статусом `inactive` или `disbanded` открывается в режиме только для чтения, поле ввода скрыто.
