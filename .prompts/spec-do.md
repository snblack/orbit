# Spec Do

Сделай, пожалуйста, `Spec` в концепции выполнения задачи, когда мы создаем 3 документа для реализации задачи: `Brief -> Spec -> Implementation Plan`.

Ниже шаблон минимальной спеки:

```markdown
# [Название фичи]

## Цель
[Одно предложение — зачем]

## Scope
- Входит: [что делаем]
- НЕ входит: [что не делаем]

## Требования
1. [Конкретное требование]
2. [Конкретное требование]

## Инварианты
- [Условие, которое должно оставаться истинным всегда, до и после реализации]
- [Пример: «Баланс счёта >= 0», «Все запросы проходят через auth middleware»]

## Acceptance Criteria
- [ ] [Проверяемый критерий]
- [ ] [Проверяемый критерий]
- [ ] Все существующие тесты проходят
- [ ] Инварианты не нарушены

## Ограничения
- [Что НЕ менять, какие паттерны использовать]
```

Вот brief:

```md
[вставь сюда brief]
```

## Что нужно сделать

- На основе brief подготовь spec.
- Если brief недостаточен, сначала задай минимальный список уточняющих вопросов.
- Не превращай spec в implementation plan.
- Требования и acceptance criteria должны быть конкретными и проверяемыми.
- Укажи scope явно: что входит и что не входит.
- Явно перечисли инварианты и ограничения реализации.

## Ожидаемый результат

Верни готовую spec в Markdown, пригодную для сохранения в репозитории.
