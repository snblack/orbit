# Brief Do

Сделай, пожалуйста, `Brief` в концепции выполнения задачи, когда мы создаем 3 документа для реализации задачи: `Brief -> Spec -> Implementation Plan`.

## Что такое Brief

`Brief` — это входная точка задачи. Это может быть GitHub Issue, тикет, сообщение от стейкхолдера или другой исходный запрос, переведенный в явный документ.

`Brief` фиксирует проблему и намерение, но не описывает решение.

## Brief отвечает на вопросы

- Какую проблему решаем? Боль пользователя, инцидент или бизнес-потребность.
- Для кого решаем? Какой пользователь, стейкхолдер или команда затронуты.
- Откуда взялась задача? Какой у нее контекст происхождения.
- Что хотим получить? Как выглядит желаемый результат на уровне намерения.

## Пример brief

- Проблема: при регистрации нового пользователя нет подтверждения email. Любой может указать чужой адрес и получить доступ. Поддержка получает 5-10 обращений в неделю от пользователей, чьи адреса были использованы без их ведома.
- Для кого: пользователи, которых это затрагивает; команда поддержки, которая обрабатывает обращения.
- Желаемый результат: после регистрации пользователь подтверждает email до получения доступа к системе.

## Что нужно сделать

Создай `brief.md` по пути `memory-bank/features/{xxx}/brief.md` для этой задачи:

```text
[вставь сюда описание задачи, issue, запрос стейкхолдера или контекст]
```

## Требования к результату

- Не предлагай архитектурное решение и не уходи в implementation details.
- Сфокусируйся только на проблеме, контексте, affected users/stakeholders и желаемом результате.
- Если данных недостаточно, сначала перечисли короткий список уточняющих вопросов.
- Если данных достаточно, сразу подготовь готовый текст `brief.md`.
